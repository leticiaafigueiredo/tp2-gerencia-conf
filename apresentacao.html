Jenkins

Jenkins — Visão Geral & Arquitetura

⚙️
O que é o Jenkins no projeto
✦ Servidor de Integração e Entrega Contínua (CI/CD)
✦ Executa builds automáticos a cada git push na main
✦ Pipeline configurado como código no Jenkinsfile
✦ Mantém histórico de builds, logs e artefatos
✦ Roda em container Docker via docker-compose.jenkins.yml
🔗
Integração GitHub → Jenkins
✦ GitHub dispara webhook a cada push na main
✦ ngrok cria túnel HTTPS: URL pública → localhost:8080
✦ Jenkins recebe notificação e inicia pipeline
✦ Script scripts/setup-ngrok.sh --webhook automatiza a configuração
📐
Pipeline as Code — por que importa para GCS
✦ O Jenkinsfile é um IC versionado no Git — a definição do pipeline evolui junto com o código
✦ Qualquer membro da equipe pode inspecionar e alterar o pipeline
✦ Mudanças no pipeline passam pelo mesmo fluxo de controle de mudanças (CCB)
✦ Rastreabilidade: cada build vinculado ao commit exato que o originou
Trigger
githubPush()
Webhook automático, sem polling
Rastreabilidade
IMAGE_TAG = BUILD_NUMBER
Artefato Docker vinculado ao build
Ambiente
venv isolado
Cada build recria o ambiente Python
Resultado
SUCCESS / FAILURE
Gate automático de qualidade

Stage: Build
✦ Cria venv Python isolado
✦ Instala dependências de requirements.txt
✦ Executa make build (verificação de sintaxe)
✦ Falha aqui = problema de dependência ou sintaxe
Stage: Test (paralelo)
✦ Unit: valida services e repos isoladamente
✦ Integration: testa endpoints via httpx
✦ Rodam simultaneamente — reduz tempo de CI
✦ Gera relatório de cobertura (pytest-cov)
Stage: Acceptance
✦ Fluxo real: cadastrar livro → usuário → emprestar → devolver
✦ Valida o comportamento do sistema como um todo
✦ Gate final antes do deploy
✦ Falha aqui = regressão funcional
Stage: Deploy
✦ docker build --target production
✦ Para container anterior se existir
✦ Sobe nova versão com docker run
✦ Health check: GET /api/v1/health

Jenkins

Integração GitHub → Jenkins via ngrok

Fluxo Completo — Webhook
1. git push origin main
Desenvolvedor empurra código para o repositório GitHub
2. GitHub dispara webhook
Requisição HTTP POST para a URL pública do ngrok
3. ngrok encaminha para localhost
Túnel seguro HTTPS → localhost:8080 (Jenkins)
4. Jenkins inicia o pipeline
Trigger githubPush() — clona repositório e executa stages
5. Resultado → histórico Jenkins
SUCCESS = release · FAILURE = hotfix obrigatório
🌐
Por que ngrok?
✦ Sem servidor cloud — Jenkins roda na máquina local
✦ ngrok cria URL pública HTTPS que tunela para localhost
✦ GitHub consegue entregar webhooks sem acesso direto à rede local
✦ Solução gratuita e funcional para contexto acadêmico

Retrospectiva

Lições Aprendidas

01
Pipeline as Code transforma o Jenkinsfile em IC

Versionar o Jenkinsfile no Git nos forçou a tratar o pipeline com o mesmo rigor do código — mudanças passam pelo CCB e têm rastreabilidade completa.

02
ngrok exige disciplina operacional

O túnel deve estar ativo antes de qualquer demo. Aprendemos a automatizar com scripts e a manter o token válido — uma lição de confiabilidade de infraestrutura.

03
Testes paralelos exigem isolamento de estado

Rodar Unit ∥ Integration no Jenkins economizou tempo, mas exigiu que cada teste crie e destrua seu próprio estado — boa prática que se pagou depois.

04
Gate automático (SUCCESS/FAILURE) exige commits pequenos

Aprovar o merge automaticamente pelo pipeline forçou commits menores e testes locais antes do push — a main ficou estável durante todo o projeto.

05
Gerência de Configuração é processo, não só ferramenta

Jenkins e Git são meios, não fins. O que realmente funcionou foi o fluxo: CCB, baselines formais e controle de mudanças com aprovação explícita.
